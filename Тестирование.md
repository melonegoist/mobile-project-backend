# Документ тестирования проекта BrokerApp

## 1. Введение

### 1.1. Назначение документа

Настоящий документ описывает подход, инструменты, тестовое окружение,
покрытие тестами и порядок запуска тестов системы **BrokerApp** —
учебной микросервисной платформы биржевого брокера, реализующей
аутентификацию пользователей, торговлю акциями, потоковую доставку
котировок и нагрузочное тестирование собственной инфраструктуры.

### 1.2. Объект тестирования

Под объектом тестирования понимается совокупность следующих сервисов:

| Сервис              | Язык / Стек                        | Назначение                                                |
| ------------------- | ---------------------------------- | --------------------------------------------------------- |
| `db-handler`        | Kotlin 2.2 / Spring Boot 4         | Бизнес-логика, REST API, JWT, JPA, потребитель Redis      |
| `mobile-handler`    | Kotlin 2.3 / Ktor 3.4              | Реверс-прокси, WebSocket-сервер, рассылка котировок       |
| `quotation-receiver`| Go 1.21                            | Чтение бинарных кадров котировок с устройства, паблиш в Redis |
| `load-tester`       | Kotlin 2.2 / Spring Boot 4         | Эмуляция нагрузки на mobile-handler через виртуальных пользователей |

### 1.3. Связанные документы

- `SRS проекта.pdf` — Software Requirements Specification.
- `docker-compose.yml` — конфигурация рабочего окружения.

## 2. Цели и задачи тестирования

### 2.1. Функциональные цели

1. Проверить корректность ключевой бизнес-логики:
   торговые операции, расчёт портфеля, аутентификация, watchlist.
2. Гарантировать поведение системы на негативных сценариях
   (недостаточно средств, продажа отсутствующих акций, дубликаты, истёкшие токены).
3. Проверить интеграцию между сервисами по реальным каналам
   (PostgreSQL, Redis pub/sub, WebSocket, HTTP).

### 2.2. Нефункциональные цели

1. Обеспечить воспроизводимость тестов: окружение поднимается автоматически
   из контейнеров, без зависимости от локально установленных Postgres/Redis.
2. Изоляция тестов: каждый тест создаёт своего пользователя и работает с
   ним; общее состояние БД не приводит к коллизиям.
3. Скорость: unit-тесты выполняются без I/O за миллисекунды;
   E2E используют singleton-контейнеры, чтобы старт Docker происходил один раз на JVM.

---

## 3. Стратегия тестирования

Применяется классическая «пирамида тестирования»:

```
                    ┌──────────────────┐
                    │   E2E (4 теста)  │  ←  реальные Postgres + Redis (Testcontainers)
                    ├──────────────────┤
                    │ Интеграционные   │  ←  Ktor testApplication + Redis-контейнер
                    │   (внутри E2E)   │
                    ├──────────────────┤
                    │  Unit (~75 кейсов)│ ←  MockK / Spring ReflectionTestUtils / go test
                    └──────────────────┘
```

**Уровни:**

- **Unit-тесты** — изолированно проверяют сервисный слой каждого
  микросервиса с подменой зависимостей моками. Не используют сеть и БД.
- **E2E-тесты** — поднимают полноценное Spring/Ktor-приложение, прогоняют
  запросы по реальному HTTP/WS-протоколу и проверяют конечное состояние
  в БД и сетевых каналах.

## 5. Тестовое окружение

### 5.1. Требования к стенду

- Docker Desktop (для E2E-тестов через Testcontainers).
- JDK 21+ для `mobile-handler`, JDK 24 для `db-handler` и `load-tester`.
- Go 1.21+ для `quotation-receiver`.

### 5.2. Изоляция

- **Unit-тесты** не выходят за границы JVM/процесса.
- **E2E-тесты** используют изолированные эфемерные контейнеры
  PostgreSQL 16 и Redis 7. Контейнеры запускаются один раз на JVM
  (singleton-pattern), миграции Flyway применяются автоматически
  при первом старте контекста.
- Каждый E2E-тест регистрирует **уникального пользователя**
  (`prefix + System.nanoTime()`), что исключает интерференцию между
  тестами в общей БД.

### 5.3. Конфигурация

Тестовые свойства передаются через `@DynamicPropertySource` (db-handler)
и `MapApplicationConfig` (mobile-handler). Это позволяет на лету подменить:

- `spring.datasource.url` → URL Postgres-контейнера
- `spring.data.redis.host` / `port` → Redis-контейнер
- `jwt.secret` → детерминированный тестовый секрет
- `quotation.redis.channel` → `quotes.ticks`

## 7. Запуск тестов

### 7.1. Все тесты сервиса

```powershell
# db-handler
.\db-handler\gradlew.bat -p db-handler test

# mobile-handler
.\mobile-handler\gradlew.bat -p mobile-handler test

# load-tester
.\load-tester\gradlew.bat -p load-tester test

# quotation-receiver
cd quotation-receiver
go test ./...
```

```powershell
.\db-handler\gradlew.bat -p db-handler test `
  --tests "*ServiceTest" --tests "*ProviderTest" `
  --tests "*ListenerTest" --tests "*UtilsTest"

.\mobile-handler\gradlew.bat -p mobile-handler test `
  --tests "*ManagerTest" --tests "*ExtractorTest"
```

### 7.3. Только E2E (требуется запущенный Docker Desktop)

```powershell
.\db-handler\gradlew.bat -p db-handler test --tests "*E2eTest"
.\mobile-handler\gradlew.bat -p mobile-handler test --tests "*E2eTest"
```

## 9. Критерии успешного тестирования

1. **Все unit-тесты проходят** на чистой машине без локально установленных
   Postgres/Redis за время **< 30 секунд** на каждый сервис.
2. **Все E2E-тесты проходят** при наличии Docker Desktop за время
   **< 3 минут** на сервис (включая загрузку и старт контейнеров на первом запуске).
3. **Полнота покрытия бизнес-сценариев**: для каждой пользовательской
   истории из SRS существует как минимум один соответствующий E2E-тест.

## 10. Известные ограничения и риски

| Ограничение | Митигация |
|-------------|-----------|
| E2E-тесты требуют Docker Desktop. На машинах без Docker они не запустятся. | Unit-тесты остаются доступны и покрывают основную бизнес-логику. |
| `mobile-handler/QuoteWebSocketE2eTest` использует `application.environment.config`, но `RedisSubscriber.start` сперва смотрит в `System.getenv("REDIS_HOST")`. Если эта переменная установлена в среде разработчика, она перекроет тестовую конфигурацию. | Перед запуском теста следует убедиться, что `REDIS_HOST`/`REDIS_PORT`/`REDIS_CHANNEL` не выставлены глобально. |
| Тесты на `LoadTestService` обращаются к адресу `http://127.0.0.1:1` (заведомо недоступному), чтобы виртуальные пользователи безопасно завершались. Это не покрывает реальный сетевой обмен. | Реальный обмен проверяется при ручном запуске нагрузочного теста против запущенной системы. |
| Нагрузочный сервис `load-tester` тестируется только структурно (метрики, конфиг, жизненный цикл). Полные нагрузочные сценарии — отдельная задача. | Покрывается ручным запуском по `docker-compose --profile loadtest up`. |
| Покрытие нефункциональных требований (производительность, отказоустойчивость) — задача отдельного нагрузочного и отказоустойчивого тестирования. | Вне области автоматизированных тестов. |

## 11. Заключение

Реализованная стратегия тестирования закрывает:

- **бизнес-логику** всех четырёх сервисов через unit-тесты с моками
  внешних зависимостей;
- **интеграцию между сервисами** через E2E-тесты с реальными
  контейнерами Postgres и Redis и реальными HTTP/WebSocket-протоколами;
- **негативные сценарии и граничные условия**: недостаточно средств,
  дубликаты, истёкшие токены, попытки доступа без авторизации;
- **критический путь распространения котировок**:
  внешний источник → Redis → db-handler (БД) и параллельно
  Redis → mobile-handler → WebSocket-клиент.

Тесты воспроизводимы, изолированы от внешней среды и могут запускаться
